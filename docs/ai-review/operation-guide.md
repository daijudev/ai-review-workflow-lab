# AIレビュー運用ガイド

このドキュメントは、PR前にAIセルフレビューを実施するための運用ガイドです。
AIを一次レビュアーとして活用し、人間レビュアーが本質的な判断に集中できる開発プロセスを目指します。

---

## なぜAIセルフレビューを導入するのか

### 導入前の課題

- レビュー待ち時間が長く、マージまでに数日かかることがある
- レビュアーが毎回同じ指摘（型安全性・テスト漏れ・命名）をしている
- 実装者が「これでいいか不安」なままPRを出してしまう
- レビュー品質が担当者のスキルや体調に依存する
- 手戻りが多く、レビュー往復が3回以上になることがある

### 導入後に期待すること

- 機械的に検出できる問題はPR前に実装者自身が修正できる
- レビュアーは仕様・設計・保守性・チームコンテキストの判断に集中できる
- レビュー品質の属人化を防ぐ（誰でも同じ観点でセルフチェックできる）
- レビュー往復回数を削減し、マージまでの時間を短縮する

---

## AIに任せる観点

AIは以下のような**機械的・客観的に判断できる観点**が得意です。

| 観点 | 具体例 |
|------|--------|
| 型安全性 | `any` の使用・型アサーションの妥当性・null 安全性 |
| コーディング規約 | 命名の一貫性・不要なコメント・デッドコード |
| テスト漏れ | ハッピーパスのみ・エラーケース未検証・境界値テスト不足 |
| コンポーネント責務 | 1コンポーネントが複数の責務を持っていないか |
| パフォーマンス | 不要な再レンダリング・useEffect の依存配列 |
| アクセシビリティ | ARIA属性の不足・フォームラベルの欠如・キーボード操作 |
| エラーハンドリング | 例外の握り潰し・エラー状態の未処理 |
| 重複コード | コピペコード・抽象化の機会 |

### AIが苦手な観点

以下は**AIの判断を鵜呑みにしないこと**。

- ビジネス仕様の正しさ（AIは仕様書を知らない）
- チームの暗黙知・過去の経緯
- 他のPRとの依存関係・タイミング
- パフォーマンスのトレードオフ判断（計測値がないと分からない）
- セキュリティ設計の妥当性（表面的な指摘しかできないことが多い）

## 人間レビュアーが見る観点

AIにセルフレビューを任せた分、レビュアーは以下に集中してください。

| 観点 | 具体的な確認事項 |
|------|---------------|
| 仕様の正しさ | 要件・チケット・仕様書と照合して実装が正しいか |
| 設計判断 | この設計でスケールするか・他の選択肢と比べてどうか |
| 保守性 | 6ヶ月後の自分が読めるか・説明できるか |
| チームコンテキスト | 既存実装との一貫性・チームのスタイルガイドとの整合 |
| リリースリスク | 本番環境への影響・他機能への副作用 |
| セキュリティ設計 | 認可・認証・データの取り扱い |

## PR前の作業フロー

```
実装完了
    ↓
【1】セルフチェック（review-checklist.md を参照）
    ↓
【2】AIセルフレビューの実施
    - docs/ai-review/prompts/ のプロンプトを使用
    - basic-pr-review.md は必ず実施
    - 変更内容に応じて frontend / test / accessibility を追加
    ↓
【3】AI指摘の整理（3分類）
    - 修正する → その場で修正
    - 人間に相談する → PRテンプレートの「レビュアーへのお願い」に記載
    - 対応しない → 理由を PRテンプレートに記載
    ↓
【4】PRテンプレートの記入
    - AIセルフレビュー結果を記載
    - 人間レビュアーに見てほしいポイントを明記
    ↓
【5】PRを作成してレビュー依頼
```

## AI指摘の3分類

AIの指摘をすべて対応する必要はありません。以下の基準で分類してください。

### ✅ 修正する

- 明らかにバグになりうるもの
- 型安全性の問題
- テストの抜け漏れ
- アクセシビリティの問題

**対応方法**: その場で修正し、PRテンプレートの「AI指摘を受けて修正した点」に記載する。

### 🤔 人間に相談する

- 仕様に関わる判断が必要なもの
- 設計トレードオフが絡むもの
- チームの方針が不明なもの

**対応方法**: PRテンプレートの「人間レビュアーに重点的に見てほしい点」に記載する。

### ❌ 対応しない（理由を明記）

- このPRのスコープ外のリファクタリング提案
- プロジェクトの方針と合わない提案
- AIが文脈を読み違えた指摘

**対応方法**: PRテンプレートの「AIから指摘されたが対応しなかった点」に理由を記載する。

> **重要**: 「対応しない」という判断自体は問題ありません。
> ただし、**理由を明記しないこと**は問題です。
> レビュアーが「なぜ対応しなかったのか」を確認しやすくするために、必ず理由を残してください。

## 注意事項

### AIの指摘を鵜呑みにしない

AIは「一次レビュアー」です。AIが「問題ない」と言っても、それは最終判断ではありません。
特に以下の場合は、AIの判断を過信しないでください。

- セキュリティに関する判断
- ビジネスロジックの正しさ
- パフォーマンスの計測が必要な最適化

### 最終責任は人間が持つ

AIレビューはあくまでサポートツールです。
コードの品質・動作・リリース判断の最終責任は、実装者とレビュアーが負います。

### AIに渡す情報を精査する

機密情報・個人情報・APIキーをAIに渡さないでください。
レビュー対象のコードに機密情報が含まれる場合は、人間レビュアーのみに依頼してください。

## 失敗パターン

### パターン1: AIの指摘を全部修正する

AIは多くの指摘を出すことがありますが、すべてが適切とは限りません。
「AIが言ったから」という理由だけで修正するのは避けてください。
チームの判断・仕様・コンテキストを踏まえた上で取捨選択してください。

### パターン2: AIレビューをPRテンプレートに書かない

せっかくAIセルフレビューを実施しても、結果をPRに残さなければレビュアーに伝わりません。
AIの指摘・対応・非対応の理由を必ずPRテンプレートに記載してください。

### パターン3: AIを使えばレビューが不要と思う

AIは人間レビュアーを不要にするツールではありません。
仕様・設計・保守性・チームコンテキストは、AIには判断できません。
AIレビューは「人間レビュアーの時間を有効活用するため」のものです。

### パターン4: 毎回同じプロンプトしか使わない

`basic-pr-review.md` だけでは観点が不足することがあります。
変更内容に応じて `frontend-review.md` / `test-review.md` / `accessibility-review.md` を追加してください。

# レビュー観点チェックリスト

PRを出す前に、このチェックリストを使ってセルフチェックを行ってください。
AIセルフレビューと組み合わせて使うことで、よりカバレッジの高いレビューができます。

---

## 使い方

1. このチェックリストをコピーして、PRコメントや手元のメモに貼り付けてください
2. 各項目を確認し、該当しない項目はスキップしてください
3. 自信がない項目は、人間レビュアーへの質問としてPRテンプレートに記載してください

---

## チェックリスト

### 1. 仕様漏れ

- [ ] チケット・仕様書に書かれた要件をすべて実装しているか
- [ ] エッジケース（入力が空・最大値・最小値）を考慮しているか
- [ ] エラー時のユーザー体験を設計・実装しているか
- [ ] 未ログイン・権限なし時の挙動を考慮しているか

### 2. 型安全性

- [ ] `any` を不必要に使っていないか
- [ ] 型アサーション（`as`）を使っている場合、その理由は正当か
- [ ] `null` / `undefined` の可能性を適切に処理しているか
- [ ] union type や discriminated union を適切に使えているか
- [ ] API レスポンスの型定義は正確か（`unknown` から始めているか）

### 3. 状態管理

- [ ] 状態の持ち方は適切か（local state / context / 外部ストアの選択）
- [ ] 派生状態を `useState` で管理していないか（`useMemo` や計算値で代替できないか）
- [ ] 状態の更新が予測可能か（副作用が意図通りか）
- [ ] 不要な状態を持っていないか

### 4. loading / empty / error 状態

- [ ] データ取得中の loading 状態を表示しているか
- [ ] データが0件の場合の empty 状態を表示しているか
- [ ] エラー時のエラー状態を表示しているか
- [ ] それぞれの状態でユーザーが次のアクションを取れるか（再試行・空の場合のCTA）

### 5. コンポーネント責務

- [ ] 1つのコンポーネントが1つの責務を持っているか
- [ ] コンポーネントが肥大化していないか（目安: 100〜150行以内）
- [ ] Props が多すぎないか（目安: 5〜7個以内。多い場合はコンポーネントの分割を検討）
- [ ] propsのバケツリレーが発生していないか
- [ ] コンポーネント名はその責務を正確に表しているか

### 6. hooks 責務

- [ ] custom hooks は「状態と副作用を抽象化」する目的で作っているか
- [ ] hooks がUIの責務（JSXの生成）を持っていないか
- [ ] `useEffect` の依存配列は正確か（不足・過剰どちらもNG）
- [ ] cleanup 処理が必要な場合に実装されているか
- [ ] hooks の命名は `use` プレフィックスで始まっているか

### 7. テスト

- [ ] 新しいロジック・コンポーネントにテストを追加したか
- [ ] ハッピーパスだけでなく、エラーケース・境界値をテストしているか
- [ ] loading / empty / error 状態のテストを書いたか
- [ ] テストが実装の詳細（内部状態・実装ロジック）に依存していないか
- [ ] テストの説明（`describe` / `it`）は日本語または分かりやすい英語で書けているか

### 8. アクセシビリティ

- [ ] インタラクティブな要素（ボタン・リンク等）に適切なラベルがあるか
- [ ] フォームの入力要素に `<label>` が関連付けられているか
- [ ] 画像に `alt` 属性があるか（装飾目的の場合は `alt=""`）
- [ ] キーボードのみで操作できるか（フォーカス順序・Enter/Spaceでの操作）
- [ ] エラーメッセージが `role="alert"` または `aria-live` で通知されるか
- [ ] カラーコントラスト比が十分か（WCAG AA: 4.5:1以上）

### 9. パフォーマンス

- [ ] リストレンダリングに適切な `key` を使っているか（配列のインデックスを避ける）
- [ ] 重い計算処理を `useMemo` でメモ化しているか
- [ ] コールバック関数を `useCallback` でメモ化しているか（子コンポーネントへの props の場合）
- [ ] 不必要な再レンダリングが発生していないか
- [ ] 画像・動画の最適化（サイズ・フォーマット・遅延読み込み）を検討したか

### 10. 保守性

- [ ] 変数・関数・コンポーネントの命名は意図を正確に伝えているか
- [ ] コメントが必要な箇所に適切なコメントがあるか（「なぜ」を書く。「何を」はコードから分かる）
- [ ] マジックナンバー・マジック文字列を定数化しているか
- [ ] コードの重複がないか（DRY原則）
- [ ] 削除したコードがコメントアウトとして残っていないか

### 11. 既存実装パターンとの整合性

- [ ] プロジェクトの命名規則に従っているか
- [ ] 既存の utility 関数・hooks・コンポーネントを再利用しているか（車輪の再発明をしていないか）
- [ ] ファイル配置はプロジェクトのディレクトリ規則に従っているか
- [ ] スタイルの書き方（Tailwind / CSS Modules等）は既存と一致しているか
- [ ] エラーハンドリングのパターンが既存実装と一致しているか

---

## 判断に迷った場合

以下の場合は、PRテンプレートの「人間レビュアーへのお願い」に記載して相談してください。

- このチェックリストの項目が「該当するかどうか分からない」
- 対応すべきか分からないAI指摘がある
- 既存実装と矛盾しているが、既存実装が正しいか分からない
- 複数のアプローチがあり、どれを選ぶべきか迷っている

> 判断を曖昧にしたままPRを出すより、「ここが迷っているので相談したい」と明示する方がレビュアーに喜ばれます。

# テストコードレビュープロンプト

テストコードの品質を確認するためのプロンプトです。
「テストを書いたが、本当に十分か不安」な場合に使用してください。

---

## プロンプト

```
あなたはReact Testing LibraryとVitestを使ったフロントエンドテストの専門家です。
以下のテストコードをレビューしてください。

## テスト対象のコード（実装）

[テスト対象の実装コードを貼り付けてください]

## テストコード

[レビュー対象のテストコードを貼り付けてください]

## レビュー観点

### 1. テストカバレッジ

- [ ] ハッピーパス（正常系）をテストしているか
- [ ] エラーケース（異常系）をテストしているか
- [ ] 境界値（空文字・最大値・最小値・null・undefined）をテストしているか
- [ ] loading / empty / error 状態それぞれをテストしているか
- [ ] ユーザーインタラクション（クリック・入力・送信）をテストしているか

### 2. テストの意図

- [ ] `describe` / `it` / `test` の説明文はテストの意図を正確に伝えているか
- [ ] テストが「何を検証しているのか」が名前から分かるか
- [ ] テストが失敗した時に、どこで何が起きたか分かりやすいか

### 3. React Testing Library のベストプラクティス

- [ ] ユーザーの視点でクエリを使っているか（`getByRole` / `getByLabelText` を優先）
- [ ] 実装の詳細（内部状態・コンポーネントの構造）に依存していないか
- [ ] `getByTestId` を乱用していないか（アクセシビリティ属性を優先）
- [ ] `waitFor` / `findBy` を適切に使っているか（非同期処理のテスト）
- [ ] `act()` が必要な箇所で使われているか

### 4. テストの独立性

- [ ] 各テストが独立して実行できるか（テスト間で状態が共有されていないか）
- [ ] `beforeEach` / `afterEach` でセットアップ・クリーンアップが適切に行われているか
- [ ] モックが適切にリセットされているか

### 5. モック戦略

- [ ] モックの粒度は適切か（過剰なモックはテストの信頼性を下げる）
- [ ] 外部依存（API・タイマー・ランダム値）を適切にモックしているか
- [ ] モックが実際の実装と乖離していないか

### 6. アサーション

- [ ] アサーションは具体的で意図が明確か
- [ ] `expect(xxx).toBeTruthy()` ではなく、より具体的なマッチャーを使っているか
- [ ] 否定のアサーション（`not.toBeInTheDocument`）も適切に使っているか

## 出力フォーマット

### 追加すべきテストケース
- テストされていない重要なケースを具体的に挙げてください
- 可能であれば、テストコードの例を示してください

### テストコードの改善点
- [ファイル名:行番号] 改善内容と修正案

### テストの良い点
- このテストコードで良かった点

### まとめ
このテストの全体的な評価と、マージ前に対応すべき点を記載してください。

## 補足

- React Testing Library の「ユーザーの視点でテストする」哲学を踏まえてレビューしてください
- テストの量より質を重視してください（意味のないテストを増やすより、重要なケースを網羅する）
- 確信がない指摘には「要確認:」と前置きをつけてください
```

---

## 特定のテスト観点のみ確認する場合

### 「どのテストを追加すべきか」だけを聞く場合

```
以下の実装コードに対して、テストすべきケースを列挙してください。
現在のテストコードと比較して、不足しているテストケースを教えてください。

## 実装コード
[実装コードを貼り付け]

## 現在のテストコード
[テストコードを貼り付け]
```

---

## 使用時のポイント

- 実装コードとテストコードを両方貼ると、「テスト漏れ」の精度が上がります
- テストが通過しているかどうかも伝えると、AI が文脈を理解しやすくなります

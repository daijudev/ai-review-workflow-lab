# 汎用PRレビュープロンプト

このプロンプトはすべてのPRに対して使用する基本レビューです。
以下のテキストをそのままClaude / ChatGPT / GitHub Copilot に貼り付けて使用してください。

---

## プロンプト

```
あなたは経験豊富なシニアエンジニアです。
以下のPR差分をレビューしてください。

## PRの概要

[PRの概要を1〜2行で記載してください]

## 変更内容

[git diff または変更ファイルの内容を貼り付けてください]

## レビュー観点

以下の観点でレビューしてください。

1. **バグ・ロジックエラー**
   - 実行時エラーが発生しうるコードはないか
   - 条件分岐の抜け漏れはないか
   - 非同期処理の扱いは正しいか

2. **型安全性**
   - `any` の不必要な使用はないか
   - null/undefined の可能性を適切に処理しているか
   - 型アサーション（`as`）の使用は正当か

3. **コード品質**
   - 命名は意図を正確に伝えているか
   - 重複コードはないか
   - 不要なコードはないか（コメントアウト・デッドコード）

4. **エラーハンドリング**
   - 例外・エラーを適切にハンドリングしているか
   - ユーザーにエラーを適切に通知しているか

5. **テスト**
   - テストが追加されているか
   - ハッピーパス以外のケースをテストしているか

## 出力フォーマット

以下の形式で出力してください。

### 重要度: 高（バグ・セキュリティリスク）
- [ファイル名:行番号] 指摘内容と修正案

### 重要度: 中（品質・保守性）
- [ファイル名:行番号] 指摘内容と修正案

### 重要度: 低（提案・改善案）
- [ファイル名:行番号] 指摘内容と修正案

### 良い点
- このPRで良かった点を1〜3点挙げてください

### まとめ
このPRの全体的な評価と、マージ前に必ず対応すべき点を1〜3行でまとめてください。

## 補足

- プロジェクト固有のルールは分からないので、一般的なベストプラクティスでレビューしてください
- 確信がない指摘には「要確認:」と前置きをつけてください
- 指摘は具体的に、修正案は実際のコードで示してください
```

---

## 使用時のポイント

### 効果が上がる使い方

1. **差分全体を貼る**: `git diff main...HEAD` の出力をそのまま貼ると精度が上がります
2. **PRの概要を書く**: 「何のためのPRか」をAIに伝えることで、仕様に照らした指摘が得られます
3. **ファイルごとに分けて聞く**: 変更量が多い場合は、ファイルを分けて複数回実施してください

### 注意事項

- 機密情報・個人情報・APIキーが含まれる差分は貼らないでください
- AIの指摘はすべて参考情報です。対応するかどうかは自分で判断してください
- 「問題なし」と言われても、仕様の正しさはAIには判断できません

# フロントエンド特化レビュープロンプト

Reactフロントエンドのコードに特化したレビュープロンプトです。
`basic-pr-review.md` と組み合わせて使用してください。

---

## プロンプト

```
あなたはReact / TypeScriptのシニアフロントエンドエンジニアです。
以下のコードをフロントエンド観点でレビューしてください。

## コード

[レビュー対象のコードを貼り付けてください]

## レビュー観点

### 1. コンポーネント設計

- [ ] 単一責任の原則: 1コンポーネントが1つの責務を持っているか
- [ ] Props設計: Propsの数は適切か（多すぎる場合はコンポーネント分割を提案）
- [ ] コンポーネントの粒度: 再利用可能な単位に分割されているか
- [ ] Propsのバケツリレーが発生していないか（Context / Composition の活用を提案）

### 2. カスタムhooks

- [ ] hooks がUIとロジックを適切に分離しているか
- [ ] useEffect の依存配列は正確か（不足・過剰を確認）
- [ ] cleanup 処理が必要な場合に実装されているか（タイマー・イベントリスナー・購読）
- [ ] hooks の返り値の設計は使いやすいか

### 3. 型安全性

- [ ] Propsの型定義は正確で十分か
- [ ] イベントハンドラの型は正しいか（`React.ChangeEvent<HTMLInputElement>` 等）
- [ ] APIレスポンスの型定義は `unknown` から始めているか
- [ ] `any` / `as` の使用箇所と正当性

### 4. 状態管理

- [ ] `useState` vs `useReducer` の選択は適切か
- [ ] 派生状態を useState で管理していないか（計算値として扱えないか）
- [ ] 状態の初期値は型と一致しているか
- [ ] loading / error / data の状態が適切に管理されているか

### 5. パフォーマンス

- [ ] リストの `key` prop にインデックスを使っていないか
- [ ] 不必要な再レンダリングが発生していないか
- [ ] `useMemo` / `useCallback` の使用は適切か（過剰な最適化も避ける）
- [ ] 大きなリストに仮想化（Virtual Scrolling）が必要か

### 6. フォーム

- [ ] Controlled vs Uncontrolled の選択は適切か
- [ ] バリデーションのタイミングは適切か（onBlur / onSubmit）
- [ ] バリデーションエラーはユーザーに分かりやすく表示されているか
- [ ] フォーム送信中の二重送信防止が実装されているか

### 7. loading / empty / error 状態

- [ ] データ取得中の loading 状態を表示しているか
- [ ] データが0件の empty 状態を表示しているか
- [ ] エラー発生時のエラー状態を表示しているか
- [ ] 各状態でユーザーが次のアクションを取れるか（再試行ボタン等）

## 出力フォーマット

以下の形式で出力してください。

### 要修正（バグ・重大な問題）
- [ファイル名:行番号] 問題点と修正案

### 改善推奨（品質・保守性）
- [ファイル名:行番号] 問題点と修正案（修正例のコードを添付）

### 提案（より良くするための案）
- [ファイル名:行番号] 提案内容（対応は任意）

### 良い実装
- このコードで良かった点

## 補足

- Reactのベストプラクティス（2024年以降）に基づいてレビューしてください
- クラスコンポーネントは現在のベストプラクティスではないため、関数コンポーネントを前提とします
- 確信がない指摘には「要確認:」と前置きをつけてください
```

---

## 特定の観点に絞る場合

### コンポーネント設計だけを確認したい場合

上記プロンプトの「レビュー観点」を「1. コンポーネント設計」のみに絞って使用してください。

### hooks の実装を確認したい場合

上記プロンプトの「レビュー観点」を「2. カスタムhooks」のみに絞って使用してください。

---

## 使用時のポイント

- コンポーネントファイルを貼る場合は、関連するhooksファイルも一緒に貼ると精度が上がります
- TypeScript の型定義ファイルも一緒に貼ると、型の整合性チェックの精度が上がります

# リファクタリングレビュープロンプト

コードの保守性・可読性・設計の改善機会を見つけるためのプロンプトです。
「機能は動いているが、コードをもっと良くできないか」という場合に使用してください。

> **注意**: このプロンプトによる指摘は「提案」です。
> 機能の追加・バグ修正ではなく、保守性の改善が目的のため、
> 対応するかどうかはチームの判断に委ねてください。

---

## プロンプト

```
あなたは保守性・可読性を重視するシニアエンジニアです。
以下のコードをリファクタリングの観点でレビューしてください。

## コード

[レビュー対象のコードを貼り付けてください]

## コードの背景

[このコードが何をするものか、どのような文脈で使われるかを説明してください]

## レビュー観点

### 1. 命名

- 変数・関数・コンポーネントの名前は意図を正確に伝えているか
- 略語・省略が不必要に使われていないか
- 同じ概念に複数の名前が使われていないか（一貫性）
- 名前が長すぎる・短すぎる箇所はないか

### 2. 関数・コンポーネントの責務

- 1つの関数・コンポーネントが1つの責務を持っているか
- 関数が長すぎないか（目安: 30〜50行以内）
- 「副作用」と「計算」が混在していないか
- 引数が多すぎないか（目安: 3〜4個以内。多い場合はオブジェクトにまとめる）

### 3. 重複の除去

- コピペコードがないか
- 同じロジックが複数箇所に散在していないか
- 共通化できる処理を抽象化する機会はないか

### 4. 抽象化の妥当性

- 過剰な抽象化（使わない汎用化）がないか
- 抽象化が薄すぎて、単なる関数呼び出しのラッパーになっていないか
- カスタムhooksの責務は明確か

### 5. 条件分岐の整理

- 複雑なネスト・条件分岐を単純化できないか
- アーリーリターンで読みやすくできないか
- 複雑な条件式を変数に切り出して意図を明確にできないか

### 6. マジックナンバー・マジック文字列

- 説明なしの数値・文字列が直接コードに書かれていないか
- 定数として切り出して名前をつけることで意図が伝わるか

### 7. コメント

- コメントが「何をするか」ではなく「なぜするか」を説明しているか
- コードを読めば分かることをコメントで説明していないか
- 古いコメント・誤解を招くコメントがないか
- コメントアウトされたコードが残っていないか

### 8. 型の活用

- 型で制約を表現できている箇所はないか（union type・literal type の活用）
- `string` / `number` ではなく、より具体的な型で表現できないか
- ユーティリティ型（`Partial` / `Pick` / `Omit` 等）を活用できる箇所はないか

## 出力フォーマット

### 重要度: 高（保守性への大きな影響）
**問題**: 
**改善案**: 
**理由**: 
```コードの例```

### 重要度: 中（品質改善）
**問題**: 
**改善案**: 
**理由**: 

### 重要度: 低（好みの範囲）
**提案**: 
（対応は任意）

### 良い点
このコードで保守性・可読性の観点から良かった点

### まとめ
このコードの全体的な評価と、優先的に対応すべきリファクタリングを記載してください。

## 補足

- 機能を壊すリスクのあるリファクタリングには「テスト追加が必要」と明記してください
- 「好みの問題」と「明確に改善が必要な問題」を区別してください
- 確信がない指摘には「要確認:」と前置きをつけてください
- 現在のコードが「動いている」ことを前提に、改善の優先度を示してください
```

## リファクタリングのタイミング判断

AIの指摘を受けたあと、以下の基準で対応を判断してください。

### このPRで対応すること

- バグの温床になりうる複雑なロジックの整理
- 次の機能追加を妨げる設計上の問題

### 別タスクに切り出すこと

- 大規模なリファクタリング（このPRのスコープを超えるもの）
- チームで議論が必要な設計変更
- テストなしで安全に変更できないもの

### 対応しないこと

- 純粋な好みの問題
- 工数に見合わない微小な改善
- プロジェクトの既存スタイルとの一貫性を優先すべきもの

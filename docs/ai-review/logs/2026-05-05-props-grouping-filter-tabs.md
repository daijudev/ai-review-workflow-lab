### AI指摘を受けて修正した点

1. **`aria-current` の型修正**（`ReviewRequestList.tsx` L99）
   `'true'`（文字列）→ `true`（boolean）。`isActive || undefined` に変更

2. **`StatusCounts` 型を `Record` 型に変更**（`ReviewRequestList.tsx` L5）
   `interface StatusCounts { all, pending, ... }` → `type StatusCounts = Record<ReviewStatus | 'all', number>`
   新しいステータスが追加されたときにコンパイルエラーで検出できるようになった

3. **空状態の判定条件を変更**（`ReviewRequestList.tsx` L122, L130）
   `filter.counts.all === 0 / > 0` → `filter.status === 'all' / !== 'all'`
   親コンポーネントの counts 計算の正確性への依存をなくした

4. **ヘッダーの件数表示をスケルトン付きで復元**（`App.tsx` L57–L65）
   ローディング中: `animate-pulse` のグレーバーを表示
   完了後: `({requests.length}件)` を表示
   エラー時: 非表示

### 気づき・振り返り

**良かった点:**
- 型安全性の指摘（`aria-current` の boolean / `Record` 型）は実装者が見落としやすい部分で有用だった
- 空状態の判定ロジックの依存関係の問題（`filter.counts.all` 参照）はバグになりうる設計上の問題で、AIが指摘して修正できた

**改善できる点:**
- ローディング中の件数表示スケルトンはAIが「削除されている」とは気づいたが、「スケルトンで代替する」という具体的な解決策まで提示できると良かった
- `FilterTabs` 分割の判断をAIが「今のPRスコープか別か」まで踏み込んで整理できると、人間が判断しやすくなる

**AIレビューの精度について:**
- 型安全性・アクセシビリティ・設計上の依存関係の指摘は高精度
- UX判断（0件タブ表示）はトレードオフを提示できたが、最終判断は人間が必要
- `useMemo` の過剰推奨はよくあるパターン（計測値がない段階での最適化提案）

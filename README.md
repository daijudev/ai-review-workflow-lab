# ai-review-workflow-lab

> AIを個人の時短ツールとして使うのではなく、チームのPRレビュー運用に組み込んだ実践リポジトリです。

---

## このリポジトリについて

このリポジトリは、**AIセルフレビューをチームの開発プロセスに組み込む仕組み**を設計・検証した実践例です。

「AIレビューを使ってみた」ではなく、「PRテンプレート・レビュープロンプト・運用ガイド・ログテンプレートを整備し、誰でも同じ品質のセルフレビューができる運用を設計した」という成果物です。

**題材アプリ**: レビュー依頼管理アプリ（Review Request Manager）  
**技術スタック**: React + TypeScript + Vite + Tailwind CSS + Vitest

---

## なぜ作ったか

フロントエンドエンジニア兼EMとして、チームのレビュープロセスに以下の課題を感じていました。

### AIレビュー導入前の問題

| 問題 | 影響 |
|------|------|
| レビュー待ちが長く、マージまで3〜5日かかることがある | 開発サイクルが遅い |
| レビュアーが毎回同じ指摘（型安全性・テスト漏れ・命名）をしている | レビュアーのリソース浪費 |
| 実装者が「これでいいか不安」なままPRを出してしまう | 手戻りが多い |
| レビュー品質が担当者のスキルや体調に依存する | 品質のばらつき |
| 「AIを使ったほうがいい」と分かっていても、観点が属人化している | 再現性がない |

### 根本的な問題

AIをすでに使っている人はいる。しかし、**個人の裁量任せになっている**。

- 使い方が統一されていない
- AIの指摘をPRに残さないので、レビュアーが同じチェックをしている
- 「AIが問題ないと言った」が記録されず、結果的に人間レビュアーが最初からやり直す

## 導入した仕組み

```
┌─────────────────────────────────────────────────────────────┐
│                    PR前の作業フロー                          │
│                                                             │
│  実装完了                                                   │
│     ↓                                                       │
│  【1】セルフチェック                                         │
│     docs/ai-review/review-checklist.md を参照               │
│     ↓                                                       │
│  【2】AIセルフレビューの実施                                 │
│     docs/ai-review/prompts/ のプロンプトを使用              │
│     ↓                                                       │
│  【3】AI指摘の3分類                                         │
│     修正する / 人間に相談する / 対応しない（理由を残す）     │
│     ↓                                                       │
│  【4】PRテンプレートの記入                                  │
│     .github/pull_request_template.md                        │
│     AIレビュー結果・人間への質問を明記                      │
│     ↓                                                       │
│  【5】PRを作成 → レビュー依頼                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 整備したもの

| ファイル | 目的 |
|---------|------|
| `.github/pull_request_template.md` | AIセルフレビュー結果・対応・未対応・人間への質問を構造化する |
| `docs/ai-review/operation-guide.md` | 運用ルール・AIと人間の役割分担を定義する |
| `docs/ai-review/review-checklist.md` | PRごとにセルフチェックできる11観点のリスト |
| `docs/ai-review/prompts/` | AIに貼って使えるレビュープロンプト集（5種） |
| `docs/ai-review/review-log-template.md` | PRごとのAIレビュー効果を記録するテンプレート |
| `docs/case-studies/sample-pr-review.md` | 実際の運用を示すサンプル事例 |

## AIに任せる観点 vs 人間レビュアーが見る観点

| AIが見る観点 | 人間が見る観点 |
|------------|-------------|
| 型安全性（`any`・null安全性） | 仕様の正しさ（チケットとの照合） |
| テスト漏れ（境界値・エラーケース） | 設計判断（この設計でスケールするか） |
| アクセシビリティ（ARIA・フォームラベル） | 保守性（6ヶ月後の自分が読めるか） |
| コーディング規約（命名・重複・コメントアウト） | チームコンテキスト（既存実装との一貫性） |
| パフォーマンス（re-render・key属性） | リリースリスク（他機能への副作用） |
| エラーハンドリング | セキュリティ設計（認可・認証） |

> **重要**: AIは「仕様が正しいかどうか」を判断できません。最終責任は人間が持ちます。

## AIレビュー運用の流れ（具体例）

### ステップ1: プロンプトを選ぶ

`docs/ai-review/prompts/` から変更内容に合わせてプロンプトを選ぶ。

```
フォームを実装した → basic-pr-review.md + frontend-review.md + accessibility-review.md
テストを追加した  → test-review.md
リファクタリングした → refactoring-review.md
```

### ステップ2: プロンプトに差分を貼る

```bash
# PRの差分を確認
git diff main...HEAD
```

プロンプトの `[変更内容]` の部分に `git diff` の出力を貼り付けてAIに送る。

### ステップ3: AI指摘を3分類する

| 分類 | 対応 |
|------|------|
| ✅ 修正する | その場で修正 |
| 🤔 人間に相談する | PRテンプレートの「レビュアーへのお願い」に記載 |
| ❌ 対応しない | PRテンプレートに理由を記載 |

### ステップ4: PRテンプレートを埋める

`.github/pull_request_template.md` の「AIセルフレビュー結果」欄に記録する。
レビュアーは「AIが何を見たか」「何が残課題か」を把握した上でレビューできる。

## 期待される効果

### 定量的な効果（サンプル事例より）

| 指標 | 導入前 | 導入後 | 改善率 |
|------|--------|--------|--------|
| レビュー往復回数 | 平均3回 | 平均1回 | -66% |
| マージまでの期間 | 3〜5日 | 1〜2日 | -60% |
| 機械的な指摘の割合 | 高い | 低い | レビュアーが本質的な判断に集中 |

### 定性的な効果

- **実装者**: 「これで出していいか」の不安が減る。AIという壁打ち相手ができる
- **レビュアー**: 型安全性・テスト漏れ・命名の指摘時間が減り、仕様・設計に集中できる
- **チーム**: レビュー観点が属人化しなくなる。新メンバーもすぐ同じ品質でセルフレビューできる

## アピールポイント

### 1. 「AIを使う」ではなく「AIをプロセスに組み込む」

AIを個人の裁量任せにせず、プロンプト・テンプレート・ガイドとして標準化しました。  
誰でも同じ観点でセルフレビューができる再現性のある仕組みです。

### 2. AIと人間の役割分担を明確に設計した

AIには機械的チェック（型・テスト・アクセシビリティ）を任せ、  
人間レビュアーには仕様・設計・保守性の判断に集中させる設計にしています。  
AIが人間レビュアーを代替するのではなく、レビュアーの生産性を高めることを目指しています。

### 3. 「AIの指摘を鵜呑みにしない」を前提とした設計

AI指摘の3分類（修正する / 相談する / 対応しない）と、  
未対応の場合の理由記載を運用ルールとして定義しています。  
AIに判断を丸投げするのではなく、人間が最終判断する構造を維持しています。

### 4. 効果測定の仕組みがある

レビューログテンプレートで「往復回数」「AIの有用率」を記録できます。  
「AIを入れた感覚」ではなく、「データで改善を確認できる」設計です。

### 5. 実務で使える粒度

プロンプトは実際にAIに貼って使える形式で書かれており、  
PRテンプレートはそのまま `.github/pull_request_template.md` として使えます。  
「参考資料」ではなく、「すぐ動く運用設計」として作りました。

## 今後の改善案

### 短期

- [ ] AIレビューログを蓄積し、月次で有用率・往復回数を集計する
- [ ] プロンプトをチームのフィードバックで継続的に改善する
- [ ] レビュー観点チェックリストをプロジェクトに合わせてカスタマイズする

### 中期

- [ ] GitHub Actions でAIレビューを半自動化する（コメントとしてPRに追記）
- [ ] プロンプトのバージョン管理（プロンプトを変えたら結果がどう変わるかを比較）
- [ ] チームのレビューログを集計して「よくある指摘パターン」を可視化する

### 長期

- [ ] チーム固有のルール（コーディング規約・設計パターン）をプロンプトに反映する
- [ ] AIレビューの有用率が下がった際にプロンプトを見直す仕組みを作る
- [ ] 他プロジェクトへの横展開のためのテンプレートリポジトリ化
